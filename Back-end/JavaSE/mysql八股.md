## 1.MySQL 中的数据排序是怎么实现的？
### 索引
1. order by 的字段正好有索引
2. 所有字段排序的方向一致
3. 联合索引要符合最左前缀原则
4. 索引要大量回表走全表扫描+filesort
5. 优先选择覆盖索引
### filesort
1. 索引用不了走filesort
2. 用sort_buffer内存缓冲区排序，数据量小于缓冲区在内存执行，不然在磁盘中做排序
3. 内存排序分：单路排序和双路排序   磁盘排序为：归并排序
4. 单路排序：查所有select的字段排序，排完直接返回，查一次
5. 双路排序：只查排序字段和行id排序，排完再去查其他字段返回，查两次
6. max_length_for_sort_data单路/双路排序切换阈值
7. 归并排序：超过sort_buffer_size,分成多个文件做快排，最后合成一个有序大文件

```sql
EXPLAIN SELECT * FROM user ORDER BY age;
-- 如果有  Extra: Using filesort 说明filesort排序

```
## 2.详细描述一条 SQL 语句在 MySQL 中的执行过程
1. 客户端与MySQL建立TCP连接，连接器负责连接和身份验证
2. 分析器对sql做词法分析拆成token，语法分析判断是否符合语法规则，生成语法树
```sql
SELECT id, name FROM user WHERE age > 18;
-- → 拆成：
-- SELECT（关键字）
-- id（列名）
-- FROM
-- user（表名）
-- WHERE
-- age > 18（条件）
```
3. 优化器决定这条 SQL 怎么执行最快生成执行计划
4. 执行器按照计划调用存储引擎接口返回结果

## 4.MySQL 的 Change Buffer 是什么？它有什么作用？
1. 是 InnoDB 用来缓存“非唯一二级索引页的变更操作（insert,delete,update）”的一块内存区域
2. 用于延迟修改索引页，从而减少随机磁盘 I/O，
3. 索引页不在Buffer Pool，它不去磁盘都索引页，而是将变更记录放一起，当索引页被访问或后台触发时再合并
4. Change Buffer属于Buffer Pool的一部分
5. 适用于写多、读少、非唯一二级索引、多随机写 的业务，如日志、消息队列、流水、批量导入等

## 5.什么是批量数据入库？相比单条插入有什么优势？
1. 将多条数据用一条sql或者一次数据库交互插入进数据库
2. 减少建立网络连接和数据库解析，编译的开销
3. 批量插入的三种方式：JDBC原生方式，一条SQL语句拼接，ORM框架封装（MP的saveBatch或MyBatis的InsertBatch）
4. JDBC和ORM框架封装开启rewriteBatchedStatements=true能大幅提高性能
5. 为什么不默认开启rewriteBatchedStatements=true？将大量插入变成一句，不好定位问题等，适用于insert语句

## 6.什么是数据全量同步和增量同步？它们各有什么优缺点？
1. 全量同步：同步全部数据
2. 增量同步：只同步更新的数据
### 增量同步方式
1. 时间戳：根据 update_time > last_sync_time 同步变更数据<br/>
问题：删除操作同步不到（一般是逻辑删除也是update操作）；MySQL的DATETIME只能到秒，一秒内修改同步不到（用高精度）
2. 触发器法：创建触发器，将变更记录写到影子表中，同步程序读影子表，但会对源库性能有影响
3. Binlog：读取MySQL的Binlog日志   
### Canal
Canal 通过模拟 MySQL 从库订阅 binlog，实现对数据变更的实时捕获。
生产环境需重点关注表过滤、位点持久化以及 binlog 的 ROW 模式配置。

## 7.什么是游标 Cursor 分页？相比传统 LIMIT OFFSET 分页有什么优势？
1. 游标是基于上一页“最后一条数据的唯一排序值”作为游标，查询“下一批数据”。
2. 传统 LIMIT OFFSET 分页必须查出前面的数据，然后丢掉取后面的数据
### 选什么做游标
要求唯一有序，比如自增主键或者时间戳+id
### 游标的局限性
1. 不能跳页。
2. 统计不了页数

## 8.MySQL 的存储引擎有哪些？它们之间有什么区别？
MySQL 常见存储引擎包括 InnoDB、MyISAM、Memory 等，其中 InnoDB 是默认且最常用的引擎。InnoDB 支持事务、行级锁、外键、MVCC，适合高并发场景；MyISAM 不支持事务，只支持表锁，现已逐步淘汰。实际生产中绝大多数业务表都使用 InnoDB

## 9.联合索引和最左前缀原则
1. 最左前缀原则:B+树按a排序，再按b,c。只要不是顺序，或者是范围查询（<,>）,索引就失效.
2. 小优化：联合索引如果最左计数很小，然后跳过a条件从b开始，会变成索引的范围扫描
3. 联合索引回不回表看查询的列是不是只包含索引

## 10.覆盖索引
1. 查询的列都是索引
2. 索引过多会导致写入维护成本高
3. 没有回表

## 11.索引下推
1. 在扫描索引的时候，把 where 条件中索引能提前判断的条件进行判断好，减少回表
2. Extra: Using index condition
3. 针对联合索引

## 12.在 MySQL 中建索引时需要注意哪些事项？
1. 索引不能太多
2. 多读少写
3. 区分度要高
4. 排序，分组，频繁查询的字段
5. 字段内容少
### 索引失效的场景
1. 索引列上使用函数 / 运算
2. 左模糊查询，LIKE 以 % 开头
3. 联合索引不满足最左前缀
4. 联合索引中出现范围查询
5. OR 条件中有字段没索引
6. 隐式类型转换
7. 优化器认为全表扫描更快
